01 — Fondations

Qu'est-ce qu'un Actor System ?

Le actor model est un modèle mathématique de calcul concurrent où l'unité fondamentale est un actor : une entité isolée qui traite les messages un par un.

Chaque actor possède un état privé auquel rien d'autre ne peut accéder. Les actors ne partagent jamais de mémoire — ils communiquent exclusivement par passage de messages asynchrones.

  • Recevoir des messages d'une mailbox privée
  • Envoyer des messages à d'autres actors par adresse
  • Créer de nouveaux actors enfants dynamiquement
  • Mettre à jour son propre état local
  • Réagir à la mort des actors surveillés
Origine

Proposé par Carl Hewitt en 1973. Popularisé par Erlang/OTP et plus tard Akka. Rorschach apporte ce modèle au TypeScript moderne avec une sécurité de type complète.

02 — Avantages

Pourquoi le Actor Model ?

La concurrence traditionnelle repose sur la mémoire partagée et les verrous — une recette pour les race conditions, les deadlocks et les défaillances en cascade. Les actors éliminent ces problèmes au niveau architectural.

Propriété État Partagé Actor Model
Isolation des données ✗ Partagé ✓ Privé
Race conditions ✗ Possible ✓ Impossible
Portée des défaillances ✗ Crash Global ✓ Isolé
Composition ✗ Fragile ✓ Fractale
Remplacement à chaud ✗ Redémarrage requis ✓ Par actor
Observabilité ✗ Ad-hoc ✓ Native
L'isolation prévient les catastrophes

Chaque actor est une unité scellée. Un crash dans un actor déclenche un événement de supervision — son parent décide : redémarrer, arrêter ou escalader. Le reste du système continue de fonctionner.

La composabilité est fractale

Un groupe d'actors résolvant un sous-problème est identique de l'extérieur à un seul actor. Les systèmes se composent en connectant des actors, pas en réécrivant les composants internes.

La scalabilité est structurelle

Ajouter plus d'actors revient à ajouter plus d'unités isolées. Pas d'état global à synchroniser. Pas de contention de verrou. L'architecture scale horizontalement par conception.

La Backpressure est de premier ordre

Chaque mailbox peut être limitée. Lorsqu'un producteur dépasse un consommateur, le système abandonne ou signale — il n'accumule jamais silencieusement de mémoire illimitée.

03 — Abstractions Fondamentales

Anatomie d'un Actor dans Rorschach

type ActorDef<M, S> = { // Gestionnaire de messages pur handler: (state: S, msg: M, ctx: ActorContext<M>) => ActorResult<M, S> // Hooks de cycle de vie lifecycle?: (state, event, ctx) => LifecycleResult // Récupération sur erreur supervision?: 'stop' | { type: 'restart', maxRetries?: number } // Queue bornée mailbox?: { capacity?: number; overflow?: 'drop-newest' } // Snapshots d'état persistence?: PersistenceAdapter<S> // Intercepteurs interceptors?: Interceptor<M, S>[] }
// ActorResult: ce qu'un gestionnaire retourne type ActorResult<M, S> = | { state: S } // mise à jour d'état | { state: S; become: Handler<M,S> } // changement de comportement | { state: S; stash: true } // différer le message
ActorRef — Le Handle opaque

La seule façon d'interagir avec un actor. Contient un nom, une fonction send(msg) et une vérification isAlive(). Vous ne touchez jamais directement à l'état d'un actor — jamais.

Les Handlers sont des fonctions pures

État donné + message → retourne un nouvel état + effets. Pas de globales ambiantes. Pas de références mutables partagées. La même entrée produit toujours la même sortie — testable en isolation.

Changement de comportement

Un actor peut retourner become(newHandler) pour changer son comportement pour les futurs messages. Utile pour implémenter des machines à états — ex : idle → active → draining.

Stashing

Les messages peuvent être mis de côté (stashed) jusqu'à ce que l'actor soit prêt à les traiter — ex : mettre en attente les messages utilisateur en attendant le retour d'un outil.

04 — Le Système

Rorschach : Vue d'ensemble

Rorschach est un assistant personnel général construit comme un système d'actors. Son but : un substrat où chaque capacité — raisonnement, mémoire, outils, communication — est un actor isolé et supervisé.

Le système n'a pas de cœur monolithique. Tout est un plugin. Les plugins sont chargés dynamiquement et supervisés indépendamment.

Hot Reload Plugin-first Config-driven Type-safe
Philosophie de conception

"L'isolation prévient les défaillances catastrophiques. La sécurité de type prévient la corruption sémantique. Le passage de messages permet une composition illimitée."

05 — Architecture des Plugins

Écosystème de Plugins

Chaque capacité est un PluginDef — un sous-arbre d'actors autonome avec sa propre configuration, son cycle de vie et son protocole de messages. Les plugins sont chargés depuis config.json et rechargeables à chaud.

cognitive

Fournisseur LLM, boucle ReAct du chatbot, gestionnaire de session, planificateur. Le cœur de raisonnement du système.

interfaces

Serveur HTTP/WebSocket, CLI, Signal TCP. Traduit les protocoles externes en messages d'actors typés.

tools

Exécution Bash, recherche web, E/S de fichiers, vision (analyse d'images), transcription audio, analyse PDF.

memory

Knowledge Graph (Grafeo DB), consolidation épisodique, rappel sémantique, notes Zettelkasten.

observability

Journalisation structurée JSONL, traçage distribué (W3C), métriques, suivi des coûts LLM.

notebook

Notes Zettelkasten, agents de synthèse, rappels contextuels et journalisation automatisée. Transforme la connaissance en entités actives.

type PluginDef<M, S, C> = { id: string; version: string initialState: S configDescriptor: { defaults: C; onConfigChange: (cfg: C) => Partial<M> } lifecycle: (state: S, event: LifecycleEvent, ctx: ActorContext<M>) => LifecycleResult handler: MessageHandler<M, S> }
06 — Moteur de Raisonnement

Cœur Cognitif : Boucle ReAct

L'actor chatbot implémente le pattern ReAct (Reason + Act). Chaque tour de l'utilisateur déclenche une boucle : appeler le LLM, analyser sa réponse, invoquer des outils, renvoyer les résultats, répéter jusqu'à la fin.

Invocation d'Outils

Le LLM émet tool_call → le chatbot invoque l'actor de l'outil via ActorRef.send() → le résultat revient via replyTo → prochain tour LLM.

Raisonnement Itératif

Le système ne répond pas "à l'aveugle". Il évalue les résultats des outils (Observation), ajuste sa réflexion (Reason) et décide si un nouveau cycle est nécessaire pour remplir l'objectif.

Streaming

Les fragments de tokens LLM reviennent via des messages llmChunk — le chatbot les accumule et les transmet au topic de sortie, que l'interface streame au navigateur.

07 — Communication

Event Bus Pub/Sub Typé

Les actors se découplent via l'EventStream — un bus pub/sub typé. Les topics portent un type fantôme T que le compilateur impose : les éditeurs doivent émettre T, les abonnés reçoivent T. Coût nul à l'exécution.

08 — Connaissance

Mémoire & Knowledge Graph

La mémoire n'est pas une base de données greffée — c'est un sous-système d'actors de premier ordre. Chaque tour de conversation passe par la mémoire épisodique, se consolide dans un graphe de connaissances structuré et est rappelé sémantiquement avant le prochain appel LLM.

  • Mémoire Épisodique : Tours bruts enregistrés dans episodic.jsonl
  • Consolidation : Le LLM extrait les entités et relations toutes les ~30s
  • Knowledge Graph : Grafeo DB — nœuds, arêtes, segmenté par utilisateur
  • Embeddings Vectoriels : Recherche sémantique via modèle OpenAI
  • Rappel : Requête du graphe → top-N faits → injection dans le prompt
  • Zettelkasten : Notes markdown atomiques liées aux nœuds du graphe
Pourquoi un Knowledge Graph ?

La recherche vectorielle plate perd la structure. Un graphe préserve les relations : "Alice travaille avec Bob chez Acme, qui utilise l'outil X." Cela permet un raisonnement multi-sauts.

10 — Sécurité & Isolation

Auth & Multi-Tenancy

Rorschach est nativement multi-utilisateur. L'isolation n'est pas une surcouche, mais une propriété fondamentale du Session Manager et du système d'actors.

Authentification par QR Code

Flux sécurisé pour terminaux mobiles : le serveur génère un secret temporaire, le client scanne et s'enregistre. Pas de mots de passe stockés en clair.

User Store & Profils

Gestion persistante des identités, préférences et clés API par utilisateur. Chaque utilisateur a son propre espace de stockage isolé.

Isolation des Sessions

Le Session Manager agit comme un gardien. Pour chaque userId, il maintient un actor Chatbot dédié. Les messages sont routés strictement par ID, garantissant qu'aucune donnée ne fuite entre les sessions.

Sandboxing des Ressources

La mémoire (KGraph) et les logs sont préfixés par userId. Un actor dans la session A ne peut physiquement pas adresser ou lire l'état d'un actor dans la session B.

11 — Capacités Avancées

Agentic Notebook

Le plugin notebook transforme Rorschach d'un simple chat en un système de gestion de connaissances actif. Les notes ne sont pas passives ; elles sont des entités avec lesquelles les agents interagissent.

  • Note Agent : Actor spécialisé dans l'organisation et la synthèse de notes.
  • Todo & Rappels : Gestionnaire de tâches intégré avec rappels proactifs basés sur le contexte.
  • Journaling : Journal de bord quotidien automatisé capturant les insights clés.
Outils de Productivité

Les agents disposent d'outils dédiés (notes.ts, todos.ts) pour lire, écrire et lier des informations dans le Zettelkasten de l'utilisateur.

Lien avec la Mémoire

Chaque note est un nœud dans le Knowledge Graph. La création d'une note déclenche automatiquement des liens vers les entités liées, enrichissant le contexte futur de l'assistant.

12 — Échelle

Principes de Scalabilité

Rorschach est conçu pour que l'architecture ne change pas à mesure que le système grandit. Ajouter un million d'agents est la même opération qu'en ajouter un. L'arbre de supervision est fractal.

Handlers sans état

Les gestionnaires sont des fonctions pures de (état, message). Pas de globales ambiantes. L'état vit dans la boucle de la mailbox, pas dans des closures. Facile à raisonner, impossible à corrompre.

Mailboxes Bornées

Chaque actor a une capacité configurable et une stratégie de débordement (drop-newest). La Backpressure est architecturale, pas applicative.

E/S Asynchrones via pipeToSelf

Les actors ne bloquent jamais. ctx.pipeToSelf(promise, onSuccess, onFailure) résout la promesse dans la mailbox — libérant la boucle d'événements.

Recharge à chaud

Un plugin peut être remplacé sans arrêter le reste du système. Ses enfants s'arrêtent, le nouveau code est importé, un nouvel actor est lancé. Zéro temps d'arrêt.

Arrêt Gracieux

Le mode drain() de la mailbox traite les messages en attente avant de quitter. Pas de messages perdus sur un SIGTERM.

Arbres de Supervision

Les défaillances sont contenues à l'origine. Un crash d'outil n'affecte pas la mémoire. Un crash de chatbot n'affecte pas le fournisseur LLM.

L'Invariant Fractal

Une sous-arborescence d'actors est indiscernable d'un seul actor — elle a un nom, accepte des messages, produit des sorties. C'est pourquoi l'architecture scale sans refonte.

13 — Round-Trip Complet

Flux de Données en Direct

Un seul message utilisateur traverse tout le système. Chaque saut est un message d'actor typé. Chaque frontière est observable.

14 — Évolution

Directions Futures

Le substrat d'actors permet des capacités qui nécessiteraient des refontes architecturales dans des systèmes monolithiques. Ce sont des extensions naturelles, pas des refontes.

Essaims Multi-Agents

Lancement de centaines d'agents spécialisés par tâche — chercheur, critique, exécuteur — coordonnés par messagerie. Le chatbot devient un orchestrateur de haut niveau.

Déploiement Distribué

Remplacement du send() local par un transport réseau (gRPC, WebSocket). Les actors migrent vers des nœuds distants sans changer leur logique métier.

Mémoire Auto-Améliorée

Un actor dédié surveille la qualité du KGraph, identifie les lacunes et planifie des tâches de synthèse ou de recherche ciblée en autonomie.

Hiérarchies d'Objectifs

Le planificateur décompose les objectifs complexes en arbres de tâches. L'arbre de supervision EST le plan — l'échec et la replanification sont structurels.

Clusters Cognitifs

Sous-arbres optimisés pour différentes tâches : agents réactifs rapides (Haiku), raisonnement profond (Opus), exécution de code, pipelines de vision — tous composables.

Apprentissage Continu

Enregistrement des résultats → comparaison aux prédictions → feedback pour le fine-tuning. Le système apprend de son propre déploiement via la consolidation.
